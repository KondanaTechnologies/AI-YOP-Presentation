Section 01

Using LLMs in Business

Tools like ChatGPT are built for probabilistic answers — useful, plausible, close enough.
Business runs on deterministic answers — the same number, every time, traceable to a source.

The Familiar Challenges

Typical challenges we see when LLMs meet enterprise data.

🎲

Outputs vary

The same question, asked twice, returns different answers.

Root cause: No governed definitions. The model improvises meaning each time. A semantic layer eliminates variance by constraining the model to a single authoritative dictionary.
📊

Numbers don't match

AI-generated figures disagree with the dashboards the business already trusts.

Root cause: Multiple definitions of the same metric. "Revenue" means one thing in Excel, another in the database. Without a governed mapping, the model picks arbitrarily.
🔍

Answers are untraceable

No one can say where a number came from or how it was derived.

Root cause: No audit chain. The model generated an answer but didn't record its source or query path. Without a "show your work" layer, every output is unverifiable.

Section 1 · The familiar challenges

Onboarding an LLM: three things to get right.

To onboard an LLM properly, you need three things: a A handbook, a map, and a review discipline.

Pillar 1

The Handbook

A governed dictionary the model reads from. So "quarterly revenue" means one thing, not seven.

Pillar 2

The Map

A declared set of relationships between the things in your business. So the model knows that the same asset appearing under four different names across four systems is one thing.

Pillar 3

Review Discipline

The review discipline ensures determinism. So the same question returns the same answer, every time, with a traceable source.

Pillar 3: Review Discipline

A handbook and a map get you capability. Determinism is what makes it trustworthy.

📋

Schema Enforcement

The model returns structured data in a defined format. No prose in place of data. No invented fields.

🔍

Double-Check Agent

A second agent verifies every answer against the source before it reaches the user. Failed checks are held back.

🗂

Versioning

Every answer is tied to a version of the data, the handbook, and the map. Answers can be reproduced later.

The rule: If the system cannot reproduce an answer, it cannot be trusted with the next one.

Section 2 · LLM vs agent

How an Agent Uses an LLM

The LLM is the reasoning engine inside the agent. The agent wraps three things around it.

🔧

Tools

Connections to the systems where work actually happens — databases, applications, APIs, email, documents.

🧠

Memory

A record of what has been done in the current task and, where relevant, across past tasks.

🔄

A Loop

The ability to plan, act, check the result, and adjust — rather than returning a single answer and stopping.

The LLM decides what to do at each step. The agent handles how to do it.

Section 3 · Fit and misfit

Four Failure Modes

The patterns that appear when AI is applied to the wrong problem — or the right problem in the wrong way.

🪄

The Automation Illusion

Replacing a human whose judgment work was invisible.

The AI handles the visible portion. The invisible portion — the context, the exceptions, the quiet catches — is what prevented errors.
📉

The Dashboard Detour

Using AI to answer questions an existing dashboard already answered deterministically.

Reliability is traded for novelty. Adding cost and uncertainty to a solved problem.
⚖️

The Compliance Gap

Placing AI in a regulated decision path without an explainability layer.

It works until the first audit. The moment an auditor asks "why?" and the answer is "the model decided," the project is over.
🔧

Rules in Disguise

Using AI for a problem that is a fixed set of conditional rules.

A rules engine would be faster, cheaper, and fully auditable. AI is overkill here.

Good Problems and Bad Problems

The good column shares the same features every time — unstructured input, a human in the loop, a source of truth, reversible action. The bad column fails on one or more of the veto questions.

✓ Good Problems to Solve with AI
Contract Review

Flag deviations from the standard template; a lawyer reviews every flag. Unstructured input, fuzzy rules, human in the loop, every answer traces to a source.

Customer Support Triage

Classify incoming tickets by intent, route to the right queue, surface relevant history to the agent. High volume, low stakes per item, human acts on the output.

Field Engineer Knowledge Assistant

Search service manuals, past incident reports, and parts catalogues to help diagnose equipment faults. Unstructured corpus, verifiable against source documents, engineer retains judgment.

Quarterly Variance Commentary

Generate a first draft of the narrative explaining why a line item moved, with source citations for every figure. Fuzzy by nature, human edits before publication, verifiable.

✗ Bad Problems to Solve with AI
Direct Posting to the General Ledger

Read invoices and post to the GL without human review. Financial consequences, no human review, and a rules engine already does this better.

Credit Approval Decisions

Approve or deny loans using an AI model without an explainability layer. Regulatory exposure, irreversible consequences, no audit defence.

Automated Price Changes on Live Products

Adjust prices across a catalogue in real time based on demand signals. High cost of a wrong answer, reversibility is costly, fits a rules engine better.

Safety-Critical Shutdown Decisions

Use AI to decide whether to halt a production line based on sensor data. Safety consequences, no room for probabilistic answers, deterministic logic is mandatory.

Section 05

Open Source vs
Closed Source

The decision between open-source and closed-source AI is not a preference. It is an architecture and cost commitment with multi-year consequences.

The Two Categories

📦

Open-Source Agents

Example: Meta's Llama family
  • Model weights are available for download
  • Can be fine-tuned and deployed on-premises
  • Lower cost at scale, but higher setup/maintenance
  • Full control over data, compliance, and updates
  • Community-driven innovation and support
🔒

Closed-Source Agents

Example: OpenAI, Anthropic, Google
  • Model weights are not available
  • Access via API, managed by vendor
  • Higher cost per token, but lower setup/maintenance
  • Managed compliance, updates, and support
  • Faster access to latest research and features

Section 5 · The two categories

Where the Gaps Remain

The capability gap between open and closed has narrowed significantly. Three gaps still matter in regulated enterprise environments.

🎯

Agentic Precision

Multi-step tool use accuracy
Closed-source models are specifically tuned for multi-step tool use. Open-source models produce incorrect parameters more frequently. In workflows touching financial systems, this gap carries operational risk.
🛡️

Governance

Audit logs, identity, safety
Closed-source vendors include managed identity, immutable audit logs, permission scopes, and safety controls as part of the product. With open-source, this entire layer is built and maintained internally — including every update, patch, and compliance review.
⏱️

Time to Production

Days vs months
Closed-source: days to weeks. Open-source: weeks to months for anything in a regulated environment. The model has no licensing cost. The engineering required to surround it does.

Conclusion

What We Learnt

The four questions we opened with, now answered.

01

AI is a powerful tool, with real limits.

It works where inputs are unstructured, rules are fuzzy, and a human can review before action. It is the wrong tool where inputs are structured, rules are codified, or a single wrong answer carries financial, legal, or safety consequences.

02

Putting AI to work takes deliberate architecture.

A handbook that governs definitions. A map that declares relationships. A discipline that enforces determinism. Without these, the model produces answers the business cannot rely on.

03

This is a leadership question.

AI deployment is a decision about where the business commits operating expense, where governance sits, and which workflows are redesigned. These decisions do not belong in the IT function alone.

04

AI has a cost, and the trade-offs are real.

Tokens, agentic inflation, model choice, and optimisation levers all move the bill. The choice between open and closed source commits the business to a cost structure and a governance model for years.

What to do next.

1.

Assess your workflows.

Identify where the business needs deterministic, repeatable answers — and separate those from the work where approximate answers are acceptable.

2.

Audit your systems for readiness.

Check whether the data sources, definitions, and relationships an AI system would depend on are in a state to be trusted. Most are not, yet.

3.

Build the foundation layer.

The handbook, the map, the review discipline. None of this is AI-specific. All of it is prerequisite.

4.

Run small, reversible pilots.

Pick one workflow that passes the filter. Build against the architecture. Measure the outcome. Use what you learn to scope the next one.
